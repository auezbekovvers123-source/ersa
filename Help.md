# Объяснение проекта (frontend + backend)

Этот репозиторий — это **React (Vite) фронтенд** + **Express backend API**, где данные хранятся в одном файле **`server/db.json`**.

Ниже:
- что за что отвечает (по файлам),
- что именно делает каждый «элемент» (контекст, роутер, страницы),
- и самое важное: **порядок работы процесса** (какие файлы “включаются” по шагам).

---

## Как запускать (чтобы всё работало)

Обычно нужны **2 терминала**:

- **Backend API**
  - `cd server`
  - `npm install`
  - `npm start`
  - API будет на `http://localhost:3000`

- **Frontend (React)**
  - в корне проекта
  - `npm install`
  - `npm run dev`
  - Vite покажет адрес типа `http://localhost:5173`

## “Линия” работы процесса (по порядку)

### 1) Пользователь открывает сайт в браузере
1. Браузер загружает **`index.html`**
2. Внутри `index.html` есть:
   - `<div id="root"></div>` — сюда React будет рисовать интерфейс
   - `<script type="module" src="/src/main.jsx"></script>` — стартовый файл React

### 2) React запускается
1. Выполняется **`src/main.jsx`**
2. Он делает:
   - находит `#root`
   - создаёт React-приложение
   - подключает **роутер** (`BrowserRouter`)
   - подключает **авторизацию** (`AuthProvider`)
   - рендерит основной компонент **`App`**

### 3) `App` решает: пускать на сайт или показать логин
1. Выполняется **`src/App.jsx`**
2. Он берёт `user` из **`AuthContext`**
3. Если `user` **нет**:
   - монтируются **только** страницы:
     - `LoginPage` (`/login`)
     - `RegisterPage` (`/register`)
   - любой другой URL отправляется на `/login`
   - важно: **`Layout`, `HomePage`, `ProductsPage` не запускаются и не монтируются**
4. Если `user` **есть**:
   - монтируется “основной сайт”:
     - `Layout` (шапка/меню)
     - страницы внутри него (home, products и т.д.)

### 4) Пользователь логинится / регистрируется
1. Пользователь вводит email + password в **`LoginPage`** или **`RegisterPage`**
2. Страница вызывает функцию `login()` или `register()` из **`AuthContext`**
3. `AuthContext` вызывает **`apiJson()`** из `src/api.js`
4. `apiJson()` делает `fetch()` на backend:
   - `POST http://localhost:3000/login` или `POST http://localhost:3000/register`
5. Backend (Express) в **`server/server.js`**:
   - читает данные из **`server/db.json`**
   - для логина сравнивает пароль с bcrypt-хешем
   - для регистрации добавляет пользователя (пароль сохраняет как bcrypt-хеш)
   - возвращает `{ user }` **без поля password**
6. После успеха `AuthContext` сохраняет пользователя:
   - в state (`user`)
   - в `localStorage` под ключом `teamproject_user`
7. React обновляет UI:
   - `App.jsx` видит, что `user` уже есть
   - и **переключается** на “основной сайт” (монтируется `Layout` и страницы)

### 5) Выход (logout)
1. В `Layout` кнопка “Log out” вызывает `logout()`
2. `AuthContext` очищает:
   - state `user`
   - `localStorage` (`teamproject_user`)
3. `App.jsx` снова возвращается к режиму “только логин/регистрация”

## Как устроен frontend (папка `src/`)

### `src/main.jsx` — старт React-приложения
**Зачем нужен:** это точка входа. Подключает Router и AuthProvider.

**Что делает:**
- создаёт root через `createRoot(...)`
- оборачивает приложение в:
  - `BrowserRouter` — маршруты (URL → страница)
  - `AuthProvider` — глобальная авторизация
- рендерит `App`

### `src/App.jsx` — главный роутер приложения
**Зачем нужен:** хранит все маршруты и главное правило: **пока не авторизован — основной сайт не монтируется**.

**Что делает:**
- читает `user` из `useAuth()`
- если `user == null` → показывает только `/login`, `/register`
- если `user != null` → включает `Layout` и нормальные страницы

### `src/context/AuthContext.jsx` — авторизация (state + функции)
**Зачем нужен:** чтобы в любом компоненте можно было взять:
- `user` — кто сейчас вошёл
- `login()` — войти
- `register()` — зарегистрироваться
- `logout()` — выйти

**Что делает по шагам:**
- при старте читает `localStorage['teamproject_user']`
- держит `user` в состоянии React (`useState`)
- при `login/register` делает запрос через `apiJson`
- при успехе сохраняет `user` в state и `localStorage`
- при `logout` очищает всё

### `src/api.js` — маленький помощник для запросов
**Зачем нужен:** чтобы не писать один и тот же `fetch(...).then(res.json())` в каждой странице.

**Что делает:**
- берёт base URL из переменной окружения:
  - `VITE_API_URL` (если задан)
  - иначе по умолчанию `http://localhost:3000`
- `apiJson(path, init)`:
  - делает `fetch`
  - парсит JSON
  - если сервер ответил ошибкой (`!res.ok`) — кидает `Error` с текстом ошибки

### `src/components/Layout.jsx` — “рамка” сайта
**Зачем нужен:** общий каркас для авторизованной части:
- шапка
- меню
- кнопка выхода
- место, куда подставляется страница (`<Outlet />`)

**Что делает:**
- показывает навигацию (Home, Products)
- показывает email пользователя
- вызывает `logout()` по кнопке
- рендерит вложенные страницы через `<Outlet />`

### `src/pages/*` — страницы (то, что показывается по URL)

#### `src/pages/LoginPage.jsx`
**URL:** `/login` (в гостевом режиме)

**Что делает:**
- форма email + password
- на submit вызывает `login({ email, password })`
- при успехе переходит на `/` (и `App` уже пускает в основной сайт)

#### `src/pages/RegisterPage.jsx`
**URL:** `/register` (в гостевом режиме)

**Что делает:**
- форма email + password + name (optional)
- на submit вызывает `register(...)`
- при успехе переходит на `/`

#### `src/pages/HomePage.jsx`
**URL:** `/` (только после входа)

**Что делает:**
- простая домашняя страница с подсказками “как запустить API и фронт”

#### `src/pages/ProductsPage.jsx`
**URL:** `/products` (только после входа)

**Что делает сейчас:**
- заглушка/описание, какие эндпоинты API использовать для CRUD

## Как устроен backend (папка `server/`)

### `server/server.js` — Express API
**Зачем нужен:** это “сервер”, который принимает запросы от фронта:
- `/register`, `/login`
- `/users`, `/categories`, `/products`, `/orders`

**Как работает внутри:**
- читает базу из **`server/db.json`** (функция `readDb()`)
- записывает обратно в `db.json` (функция `writeDb()`)
- пароли:
  - при регистрации делает bcrypt-хеш
  - при логине делает `bcrypt.compare(...)`
- наружу **не отдаёт пароль** (функция `userPublic`)

### `server/db.json` — “база данных” в формате JSON
**Зачем нужен:** вместо настоящей базы (PostgreSQL/Mongo) тут хранение в файле.

**Что внутри:**
- `users`: пользователи `{ id, email, name, password }` (password — bcrypt-хеш)
- `categories`: категории `{ id, name }`
- `products`: товары `{ id, name, price, categoryId, description }`
- `orders`: заказы `{ id, userId, total, status }`

**Когда используется:** каждый раз, когда фронт делает запрос к API (логин/регистрация/CRUD), сервер:
- читает `db.json` → меняет данные → пишет обратно.

### `server/package.json` — зависимости backend
**Зачем нужен:** тут описано, какие пакеты нужны серверу и как его запускать.

**Что важно:**
- скрипт `npm start` запускает `node server.js`
- зависимости: `express`, `cors`, `bcryptjs`

## Что является “элементами” и как они связаны

### Router (React Router)
**Где:** `src/main.jsx` и `src/App.jsx`

**Задача:** по URL (`/login`, `/products`) показывать нужную страницу.

### Context (AuthContext)
**Где:** `src/context/AuthContext.jsx`

**Задача:** один общий источник правды про авторизацию:
- кто вошёл (`user`)
- как войти/выйти
- хранение в `localStorage`, чтобы не терять вход после обновления страницы

### Pages (страницы)
**Где:** `src/pages/*`

**Задача:** конкретные экраны (логин, регистрация, home, products).

### Layout
**Где:** `src/components/Layout.jsx`

**Задача:** общий каркас для всех страниц “внутри сайта” после авторизации.

### Backend API
**Где:** `server/server.js`

**Задача:** принять запрос, проверить/записать данные в `server/db.json`, вернуть JSON-ответ.

---

## Ещё один раз: “порядок работы” в 5 строк

1. `index.html` → подключает `src/main.jsx`
2. `src/main.jsx` → включает `BrowserRouter` + `AuthProvider` → рендерит `App`
3. `AuthProvider` → читает `localStorage` и даёт `user` всему приложению
4. `App.jsx` → если `user` нет, показывает только Login/Register; если есть — показывает Layout+страницы
5. Login/Register → через `apiJson` → запрос в `server/server.js` → чтение/запись `server/db.json` → сохранение `user` в `localStorage`

---

## Примечание про папку `ersa/` в корне
Если у вас есть папка `ersa/` внутри проекта и она выглядит как копия проекта, то это обычно **случайно созданный дубль**. Для работы приложения она не нужна (фронт берётся из `src/`, сервер из `server/`).
